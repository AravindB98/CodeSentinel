A01-01|||A01:2025 Broken Access Control|||Access control enforces that users can only act within their intended permissions. Failures typically lead to unauthorized information disclosure, modification or destruction of data, or performing a business function outside the user's limits. Common vulnerabilities include: bypassing access control checks by modifying URLs, internal application state, or HTML pages; permitting viewing or editing someone else's account by providing its unique identifier (insecure direct object reference or IDOR); elevation of privilege, such as acting as a user without being logged in or acting as an admin when logged in as a user; metadata manipulation such as replaying or tampering with JWT access control tokens. Related CWEs include CWE-22 Path Traversal, CWE-284 Improper Access Control, CWE-285 Improper Authorization, CWE-639 Authorization Bypass Through User-Controlled Key. Mitigations include deny by default, enforce record ownership at the data layer, log failures, and rate-limit API access.

A02-01|||A02:2025 Cryptographic Failures|||Previously known as Sensitive Data Exposure. Focuses on failures related to cryptography which often lead to exposure of sensitive data. Common issues include transmitting data in clear text (HTTP, SMTP, FTP), using old or weak cryptographic algorithms (MD5, SHA1, DES, RC4), using default or hardcoded keys, missing or weak certificate validation, and insufficient randomness in security contexts. Related CWEs: CWE-259 Use of Hard-coded Password, CWE-327 Use of a Broken or Risky Cryptographic Algorithm, CWE-331 Insufficient Entropy, CWE-798 Use of Hard-coded Credentials. Mitigations include classifying data, applying controls per classification, encrypting all sensitive data at rest and in transit, using strong well-known algorithms and strong keys, and avoiding storing sensitive data unnecessarily.

A03-01|||A03:2025 Injection|||An application is vulnerable to injection when user-supplied data is not validated, filtered, or sanitized, and is concatenated into commands or queries interpreted by a downstream system. Includes SQL injection, NoSQL injection, OS command injection, LDAP injection, and expression language injection. Related CWEs: CWE-20 Improper Input Validation, CWE-78 OS Command Injection, CWE-79 Cross-Site Scripting (also covered by this category in 2021+), CWE-89 SQL Injection, CWE-94 Code Injection, CWE-917 Expression Language Injection. The fix pattern for SQL injection is always parameterized queries (prepared statements) with placeholders. f-strings, string concatenation, or string formatting that merges untrusted input into a query is a direct red flag. For OS commands, avoid shell=True in Python subprocess and pass arguments as a list. For NoSQL, avoid passing raw user objects into query builders.

A04-01|||A04:2025 Insecure Design|||Focuses on risks related to design and architectural flaws. Insecure design is a broad category that includes missing threat modeling, insecure design patterns, and lack of security requirements in the SDLC. An insecure design cannot be fixed by a perfect implementation; the design itself must be revisited. Related CWEs: CWE-209 Information Exposure Through an Error Message, CWE-256 Plaintext Storage of a Password, CWE-501 Trust Boundary Violation, CWE-522 Insufficiently Protected Credentials. Mitigations include threat modeling, secure design patterns, reference architectures, and tenant segregation.

A05-01|||A05:2025 Security Misconfiguration|||Application security misconfigurations can occur at any level of the application stack. Common symptoms: missing appropriate security hardening, unnecessary features enabled (ports, services, accounts), default accounts with unchanged passwords, overly informative error messages revealing stack traces, missing security headers, outdated software. Related CWEs: CWE-16 Configuration, CWE-209 Information Exposure, CWE-611 XML External Entities (XXE), CWE-614 Sensitive Cookie Without Secure Flag. Mitigations: repeatable hardening process, minimal platform, segmented architecture, automated verification.

A06-01|||A06:2025 Vulnerable and Outdated Components|||Using components with known vulnerabilities is a widespread risk. Applications are vulnerable if the team does not know versions of all components, software is unsupported or out of date, and components are not scanned regularly. Related CWEs: CWE-1104 Use of Unmaintained Third Party Components, CWE-937 OWASP Top Ten 2013 A9. Mitigations: remove unused dependencies, continuously inventory versions, monitor sources like CVE and NVD, obtain components from official sources, monitor libraries that are unmaintained.

A07-01|||A07:2025 Identification and Authentication Failures|||Confirmation of the user's identity, authentication, and session management is critical. Common failures: permits automated attacks such as credential stuffing, permits brute force or other automated attacks, permits default/weak/well-known passwords, uses weak or ineffective credential recovery, stores passwords in plain text or using reversible encryption, has missing or ineffective multi-factor authentication, exposes session identifiers in the URL. Related CWEs: CWE-287 Improper Authentication, CWE-297 Improper Validation of Certificate, CWE-384 Session Fixation, CWE-521 Weak Password Requirements. Mitigations: multi-factor authentication, do not ship default credentials, password length and complexity checks against published breach lists, rate-limit authentication.

A08-01|||A08:2025 Software and Data Integrity Failures|||Relates to code and infrastructure that does not protect against integrity violations. An application relies on plugins, libraries, or modules from untrusted sources or repositories; an insecure CI/CD pipeline; auto-update without integrity verification; or untrusted deserialization. Includes CWE-502 Deserialization of Untrusted Data. Python pickle.loads, Java ObjectInputStream, and similar mechanisms are classic attack surfaces. Fix pattern: never deserialize untrusted input with pickle; use signed tokens (itsdangerous.URLSafeSerializer), JSON with explicit schemas, or safer formats. For CI/CD, verify code signatures and use trusted repositories.

A09-01|||A09:2025 Security Logging and Monitoring Failures|||Without logging and monitoring, breaches cannot be detected. Common: auditable events (logins, high-value transactions) not logged; warnings and errors producing no/inadequate logs; logs not monitored for suspicious activity; logs only stored locally. Related CWEs: CWE-117 Improper Output Neutralization for Logs, CWE-223 Omission of Security-relevant Information, CWE-532 Insertion of Sensitive Information into Log File. Mitigations: ensure logins and input validation failures are logged with sufficient context, logs are in a format consumable by log management tools, and monitoring and alerting are in place.

A10-01|||A10:2025 Server-Side Request Forgery (SSRF)|||SSRF flaws occur when a web application fetches a remote resource without validating the user-supplied URL. This allows an attacker to coerce the application to send a crafted request to an unexpected destination, even when protected by a firewall. Related CWE: CWE-918 Server-Side Request Forgery. Mitigations: sanitize and validate all client-supplied input data; enforce URL schema, port, and destination with a positive allow-list; disable HTTP redirections; do not send raw responses to clients; be aware of URL consistency attacks.
